# FileShare — обзор программы

Репозиторий **FileExchange**: desktop-клиент **FileShareClient** (Windows) и backend **FileShareServer**. Мессенджер и обмен файлами только между пользователями со статусом **друзья** (взаимное принятие заявки).

Подробные цепочки вызовов (REST, SignalR, файлы, диаграммы): **[ARCHITECTURE_APPLICATION_LIFECYCLE.md](ARCHITECTURE_APPLICATION_LIFECYCLE.md)**. Краткая инструкция по запуску и конфигурации: **[README.md](README.md)**.

---

## 1. Назначение

- **Realtime:** чат, онлайн-друзья, уведомления о заявках и о готовых серверных файлах — через **SignalR** (`/chathub`, JWT в query `access_token`).
- **История:** сообщения и метаданные файлов в **SQLite** (`fileshare.db` на сервере).
- **Файлы:** гибрид — **P2P** (WebRTC Data Channel, сигналинг через тот же хаб) и/или **сервер** (multipart upload, файлы на диске в `uploads/`, скачивание по `GET /api/files/download/{id}`).

---

## 2. Основные возможности

- Регистрация и вход (JWT на клиенте в памяти, **BCrypt** для паролей на сервере).
- Поиск пользователей, заявки в друзья, принятие / отклонение, удаление из друзей; список **исходящих** заявок (`GET /api/friends/sent`).
- Список друзей с **онлайн**-индикатором (по активным подключениям к хабу).
- Личный чат с выбранным другом: история, поиск по тексту сообщений, непрочитанные счётчики, пометка прочитанным через API.
- Файлы: выбор через UI, **drag-and-drop** (`chatFileDrop.js`), режимы **AUTO / P2P / SERVER** (см. §5).
- Входящие файлы, загруженные на сервер: список **`GET /api/files/inbox`** (при необходимости отдельно от чата).

---

## 3. Архитектура

### 3.1. Клиент (FileShareClient)

- **Платформа:** .NET **10** (`net10.0-windows`), окно **WinForms** + **Blazor WebView** (`wwwroot/index.html`, маршруты `/` — логин, `/chat` — чат).
- **Сеть:** `ApiService` (`HttpClient`) для REST; `ChatService` (`HubConnection`) для SignalR.
- **JS:** `peerTransfer.js` — WebRTC (offer/answer/ICE через хаб, текст и файлы по Data Channel); `chatFileDrop.js` — drop-зона и скролл; `chatComposer.js` — вспомогательная логика композера.

### 3.2. Сервер (FileShareServer)

- **Платформа:** .NET **10** (`net10.0`), **Minimal API** (группа `/api/...`), **SignalR** `ChatHub` на `/chathub`.
- **Данные:** EF Core + **SQLite**; загрузки — папка **`uploads/`**.
- **Авторизация:** JWT Bearer для HTTP; для WebSocket тот же токен передаётся query-параметром (обработка в `JwtBearerEvents.OnMessageReceived`).

---

## 4. Как работает чат

1. После логина клиент сохраняет сессию (`SetSession`: токен в заголовке `Authorization`, объект текущего пользователя).
2. На странице `/chat` вызывается **`ChatService.ConnectAsync(ServerUrl, token)`** — подключение к `/chathub?access_token=...`.
3. **Текст:**
   - если друг **онлайн**, сначала вызывается **`peerTransfer.sendText`** (P2P); при успехе в фоне **`POST /api/chat/store/{friendId}`** дублирует сообщение в БД **без** push `ReceiveMessage` из этого эндпоинта;
   - иначе (друг офлайн или P2P не сработал) — **`InvokeAsync("SendMessage", ...)`** на хабе: сервер проверяет дружбу, пишет в БД, шлёт получателю **`ReceiveMessage`**.
4. При открытии диалога подгружается история **`GET /api/chat/conversation/{friendId}`**, непрочитанные помечаются **`POST /api/chat/mark-read/{messageId}`**.
5. Счётчики непрочитанных строятся по **`GET /api/chat/unread`** и обновляются при входящих событиях / перезагрузке данных.

## 5. Как работает обмен файлами

Пользователь задаёт режим в UI (**AUTO / P2P / SERVER**).

| Режим | Поведение |
|-------|-----------|
| **SERVER** | Только `POST /api/files/upload/{receiverId}` (multipart). Лимит **100 МБ** на клиенте и на сервере (Kestrel `MaxRequestBodySize`). |
| **P2P** | Передача через **`peerTransfer.sendFileStream`**; после успеха — **`POST /api/chat/store-file/{friendId}`** с `source: p2p` и токеном; при ошибке **нет** fallback на сервер. |
| **AUTO** | Сначала P2P; при неудаче — сервер **только если** размер файла **≤ 100 МБ**; иначе пользователю показывается ошибка (серверный путь недоступен для «тяжёлых» файлов без P2P). |

### Техническая реализация

- **Сервер:** запись файла на диск, запись **`FileTransfer`** в БД, строка сообщения в чате вида **`FILE|{id}|…|server|-`**; push **`FileTransferAvailable`**, **`ReceiveMessage`**, **`SocialDataChanged`**.
- **P2P:** бинарные чанки в JS (базовый шаг **256 KiB**, для файлов **> 10 MiB** при отправке используется **512 KiB** на шаг — см. `peerTransfer.js`); ограничение SCTP учитывается (`maxSctpPayloadPerSend`).
- **Скачивание с сервера:** сервер отдаёт поток **`Results.Stream`**; клиент **`DownloadFileAsync`** читает ответ в **память** (`MemoryStream`) перед сохранением через диалог — для очень больших файлов это ограничение клиента.
- **P2P после приёма:** байты лежат в **`P2pFileCache`** в памяти процесса; скачивание из сообщения для `source: p2p` идёт из кэша (после перезапуска клиента кэш пуст).

### Отображение в UI

В списке сообщений файловое вложение помечается источником **P2P** или **SERVER** (парсинг поля `Content` сообщения типа «файл»).

## 6. Что означает P2P в текущей реализации

- **WebRTC Data Channel** между двумя клиентами после обмена SDP/ICE через **`ChatHub`** (`SendOffer` / `SendAnswer` / `SendIceCandidate` и зеркальные события на клиенте).
- **Опционально:** `InitiateFileTransfer` на хабе — уведомление о намерении передать файл (UX).
- **Метаданные в чате:** после успешной P2P-отправки запись в БД через **`store-file`**; получатель при полном приёме может обновить ленту через **`GET .../conversation`**.
- **Сигналинг всегда через сервер**; полезная нагрузка файла при P2P между клиентами **не** гоняется через ASP.NET как бинарный relay.

---

## 7. Безопасность и ограничения

- Доступ к чату, друзьям и файлам по API — только с **валидным JWT**; операции с данными друзей проверяют пару пользователей через **`FriendshipService`**.
- Пароли — **BCrypt**.
- **HTTPS** предполагается для продакшена; в коде клиента для разработки отключена проверка сертификата TLS у HTTP и SignalR — перед выкладкой наружу это нужно убрать.
- Конфигурация: **`Jwt:Secret`** (и при необходимости issuer/audience) в `appsettings.json`; URL сервера на клиенте — свойство **`ApiService.ServerUrl`**.

---

## 8. Зачем такая архитектура

- **Доставка файлов:** P2P снижает нагрузку на сервер и трафик; сервер даёт **надёжный fallback** в AUTO для файлов до 100 МБ, когда прямой канал не поднимается.
- **Единый канал событий:** один хаб закрывает чат, presence, дружбу и WebRTC-сигналинг.
- **Прозрачность:** в UI видно, прошёл ли файл через сервер или по P2P.
- **Развитие:** см. §10; детали оптимизаций — **[PERFORMANCE_IMPROVEMENTS.md](PERFORMANCE_IMPROVEMENTS.md)**.

## 9. Типовой сценарий

1. Пользователь A регистрируется или входит, открывается `/chat`, поднимается SignalR.
2. A ищет B, отправляет заявку (`POST /api/friends/request/...`); B получает push и обновляет списки.
3. B принимает заявку; оба видят друг друга в списке друзей.
4. Обмен текстом (см. §4): при онлайн возможен P2P-текст с записью в БД через `store`.
5. A отправляет файл в режиме **AUTO**: при успешном P2P — метаданные через `store-file`; иначе — upload на сервер и уведомления получателю.
6. B открывает вложение: для **SERVER** — запрос `download/{transferId}`; для **P2P** — сохранение из локального кэша, если сессия жива.

## 10. Идеи для развития

### Функциональность

- Групповые чаты и многоадресная передача файлов.
- Аудио/видео через WebRTC.
- Пауза / возобновление и chunked upload с offset для очень больших файлов.
- Сквозное шифрование сообщений (отдельный слой поверх текущего хранения).

### UX и настройки

- Показ «канал P2P готов / нет» перед отправкой большого файла.
- Единый экран настроек: URL сервера, режим файлов по умолчанию, лимиты.
- Исправление выхода из аккаунта во всех layout: везде вызывать **`ClearSession`** перед переходом на `/`.

### Надёжность и продакшен

- **Refresh tokens** вместо одного долгоживущего JWT.
- Rate limiting, аудит, типизированная валидация вложений.
- Потоковое сохранение скачиваемого файла на диск клиента без полной буферизации в RAM.
- Логирование (например Serilog), метрики, вынос SignalR за балансировщик с sticky sessions при масштабировании.
