# FileShare - Program Overview

## 1. Что это за программа

FileShare — desktop-мессенджер для общения и передачи файлов между пользователями, которые добавили друг друга в друзья.

Ключевая идея продукта:
- realtime чат,
- контролируемый обмен файлами,
- гибридная доставка: P2P и/или сервер.

---

## 2. Основные возможности

- Регистрация и вход в аккаунт.
- Поиск пользователей.
- Отправка и обработка заявок в друзья.
- Список друзей с онлайн-статусами.
- Личные сообщения в чате.
- Счетчики непрочитанных сообщений.
- Отправка файлов:
  - выбор через скрепку,
  - drag-and-drop,
  - режим передачи `AUTO / P2P / SERVER`.
- История сообщений и файлов в чате.

---

## 3. Архитектура

## Client (FileShareClient)

- WinForms окно + BlazorWebView (гибридный интерфейс).
- Главный экран чата: друзья, сообщения, файлы, статус.
- SignalR клиент для realtime событий.
- JS-модули:
  - `chatFileDrop.js` — drag-and-drop и служебные UI-хелперы,
  - `peerTransfer.js` — WebRTC DataChannel для P2P.

## Server (FileShareServer)

- ASP.NET Core Minimal API.
- SignalR Hub (`/chathub`) для realtime и WebRTC-сигналинга.
- EF Core + SQLite (`fileshare.db`).
- JWT authentication.

---

## 4. Как работает чат

1. Пользователь логинится, получает JWT.
2. Клиент подключается к SignalR Hub с токеном.
3. Сообщения отправляются в чат, отображаются в realtime.
4. На сервере сообщения сохраняются в БД как история.
5. Непрочитанные сообщения учитываются и показываются бейджами в списке друзей.

---

## 5. Как работает обмен файлами

Пользователь выбирает режим отправки:

- **AUTO**  
  Сначала попытка P2P, если не удалось — fallback через сервер.

- **P2P only**  
  Только прямой канал между клиентами (WebRTC DataChannel).

- **SERVER only**  
  Только загрузка файла на сервер и скачивание получателем (HTTP multipart).

### Техническая реализация

- **Максимальный размер**: 100 MB (проверка на сервере и Kestrel limit)
- **P2P передача**: Бинарный формат (ArrayBuffer) + 64KB чанки для оптимального баланса между throughput и памятью
- **SERVER передача**: Потоковая загрузка (File.OpenRead + Results.Stream), без загрузки файла целиком в память
- **Интеграция**: Оба режима используют SignalR для сигналинга и отслеживания статуса

### Отображение в UI

Каждое файловое сообщение в чате помечается источником:
- `P2P` — передача прошла по P2P каналу
- `SERVER` — передача прошла через сервер

Это позволяет пользователю сразу понимать, как именно файл был доставлен.

---

## 6. Что означает P2P в текущей реализации

- **WebRTC DataChannel**: Установка прямого канала между двумя клиентами
- **Бинарная передача**: Файлы передаются в виде ArrayBuffer (не Base64), что снижает overhead на передачу данных
- **Чанкирование**: Файл разбивается на 64KB блоки для эффективного использования пропускной способности
- **SignalR сигналинг**: Используется для обмена offer/answer/ICE-кандидатами при установке WebRTC соединения
- **Фиксирование результата**: В БД сохраняется информация о том, какой режим был использован (P2P или SERVER)
- **Fallback логика**: В режиме AUTO, если P2P канал не установлен за разумное время, автоматически переключается на серверный путь

---

## 7. Безопасность

- JWT для авторизации API и SignalR.
- Пароли хранятся в bcrypt-хэшах.
- HTTPS для запросов.
- Передача файлов ограничена контекстом дружбы.
- Пользователь явно выбирает режим передачи файла.

---

## 8. Почему эта архитектура удобна для продакшена

- **Надежность**: Fallback на сервер гарантирует доставку файла в любых условиях сети
- **Прозрачность**: В интерфейсе видно, каким способом доставлен файл — пользователь понимает маршрут передачи
- **Масштабируемость**: SignalR Hub и API легко выносить на отдельные сервера или масштабировать горизонтально
- **Производительность**: Потоковая обработка файлов и бинарные трансферы минимизируют потребление памяти
- **Расширяемость**: Можно добавить прогресс передачи, подтверждения доставки, проверку типов файлов, лимиты и аудит
- **Гибкость**: Пользователь может явно выбрать режим передачи или доверить системе (AUTO)

---

## 9. Типовой пользовательский сценарий

1. Пользователь A логинится.
2. Находит пользователя B и отправляет заявку.
3. Пользователь B принимает.
4. A и B общаются в чате.
5. A отправляет файл с режимом `AUTO`.
6. Если P2P доступен — файл идет напрямую; иначе сервер обеспечивает доставку.
7. B видит в сообщении источник (`P2P`/`SERVER`) и скачивает файл.

---

## 10. Идеи для развития

### Функциональность
- Полноценные групповые чаты с многосторонней передачей файлов
- Голосовые вызовы через WebRTC
- Отправка больших файлов с паузой/возобновлением передачи
- End-to-End шифрование сообщений (TweetNaCl.js)

### Пользовательский опыт
- Индикатор прогресса передачи с информацией о скорости и оставшемся времени
- Отображение статуса доступности P2P канала перед отправкой
- История передач с детализацией по типам (P2P/SERVER)
- Управление настройками: выбор режима по умолчанию, ограничения размера

### Мониторинг и надежность
- Логирование статистики передач (Serilog)
- Rate limiting для защиты от spam
- Валидация типов файлов на обеих сторонах
- Кэширование (Redis) для списков друзей и часто запрашиваемых данных
- Refresh tokens вместо долгоживущих JWT
